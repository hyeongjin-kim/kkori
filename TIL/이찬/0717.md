- `Promise가 무엇인가요?`

  `Promise` 는 비동기 작업의 완료를 나타내는 객체입니다. then, catch, finally등을 통해 비동기 흐름을 체이닝 방식으로 명확하게 관리할 수 있습니다.

  - `Promise가 왜 생겼나요?`
    비동기 작업을 중첩된 콜백 함수로 처리하면 들여쓰기가 계속되고, 코드가 복잡해져 가독성이 떨어지는 문제가 있습니다. 이를 콜백지옥이라고 하는데, 프로미스를 사용하면 일정한 들여쓰기로 가독성을 개선할 수 있습니다.
    ```jsx
    //콜백 지옥
    setTimeout(() => {
      console.log('1');
      setTimeout(() => {
        console.log('2');
        setTimeout(() => {
          console.log('3');
        }, 1000);
      }, 1000);
    }, 1000);

    //Promise로 개선
    function delayLog(msg) {
      return new Promise((resolve) => {
        setTimeout(() => {
          console.log(msg);
          resolve();
        }, 1000);
      });
    }

    delayLog('1')
      .then(() => delayLog('2'))
      .then(() => delayLog('3'));
    ```
  - `Promise.all이 무엇인가요?`
    Promise.all은 여러 개의 Promise를 병렬로 실행하고, 모두 완료되면 결과를 배열로 반환하는 메서드입니다. 하나라도 실패하면 전체가 reject됩니다. 이를 방지하기 위해 실패하는 Promise가 있더라도 나머지를 이행하는 Promise.allSettled가 있습니다.
  - `Promise.all과 Promise.allSettled는 어떻게 다른가요?`
    둘 다 여러 개의 Promise를 병렬적으로 처리하지만, 실패하는 경우를 처리하는 방식이 다릅니다.
    `Promise.all`은 내부의 모든 Promise가 성공해야 then으로 넘어가지만, 하나라도 reject되면 catch로 넘어갑니다.
    하지만 `Promise.allSettlled`는 성공 실패와 무관하게 모든 결과를 객체 형태로 반환하며
  - `Promise의 단점은 무엇인가요?`
    체이닝 방식으로 비동기 흐름을 관리하기 때문에 체이닝이 길어지면 가독성이 떨어진다는 단점이 있습니다. 이를 극복하기 위해 async, await방식을 사용합니다.
  - `async, await가 무엇인가요?`
    async, await는 Promise를 기반으로 비동기 코드를 동기식으로 읽기 쉽게 만드는 문법입니다. async 함수는 항상 Promise를 반환하고, await는 이 Promise가 해결될 때까지 기다립니다.
  - `await를 왜 써야 할까요?`
    await는 비동기 처리의 완료 시점을 명확히 하고, 동기 코드처럼 흐름을 구성할 수 있도록 도와줍니다.
    기존에는 Promise로 콜백을 쓰거나 외부 변수를 통해 처리 시점을 따로 나눠야 했지만 await 는 해당 비동기 작업이 끝날 때까지 중단(wait) 되므로 가독성과 유지보수성이 좋아집니다.
  - `async, await에서 우아하게 예외처리를 하는 방법은 어떤 것이 있을까요?`

    ```jsx
    function to(promise) {
      return promise.then((data) => [null, data]).catch((err) => [err, null]);
    }
    const [err, result] = await to(fetchData());
    if (err) {
      console.error('에러:', err);
      return;
    }
    console.log('결과:', result);
    ```

    위와 같이 then과 catch가 있는 Promise로 한 번 더 감싼 뒤, 이를 실행한 결과를 가지고 예외처리를 하면 불필요한 depth없이 할 수 있습니다.

  - 로그인 버튼을 눌렀을 때, 모달 띄우기 VS 새 창으로 이동

  모달을 띄우는 경우, 사용자가 보던 페이지를 그대로 보기 때문에 페이지 전환이 없기 때문에 사용 흐름을 해치지 않고, UI 상태를 유지할 수 있다는 장점이 있습니다.

  하지만 구현 복잡도가 증가하고, 브라우저 히스토리 관리가 까다롭고, 페이지 자체가 바뀌지 않아 SEO에 불리하다는 단점이 있습니다.

  새 창으로 이동(/login으로 이동)하는 방식은 라우팅 기반이기 때문에 관리가 편하고 구조가 명확하고 단순하다는 장점이 있습니다.

  하지만 페이지를 이동하기 때문에 이동하면서 UI 상태가 초기화되고, 화면이 전환되기 때문에 사용자 흐름을 해칠 수 있다는 단점이 있습니다.

  우리는 로그인에 특별한 상태 저장이 필요 없고, SEO와 명확한 인증 흐름을 우선시했기 때문에 페이지 이동 방식을 채택했습니다.

  우리는 자체 인증 UI를 직접 구성하지 않고, OAuth 기반의 구글 로그인을 사용할 예정이기 때문에, 로그인 화면은 단순한 redirect 버튼만으로 구성될 수 있습니다. 또한 로그인을 해야 , 인증 흐름도 단순하기 때문에,

  모달보다는 `/login` 같은 별도 페이지로 분리하는 방식이 더 명확하고 관리가 용이하다고 판단했습니다.

- `HTTP가 뭔가요?`
  HTTP는 Hyper Text Transfer Protocol로 문서나 데이터를 전달하기 위해 사용하는 통신 규약입니다.
  HTTP의 특징으로는 비연결성, 무상태성이 있습니다.
  비연결성(Connentionless)은 요청-응답 후 연결이 종료된다는 특징입니다.
  무상태성(Stateless)는 요청 간 상태를 저장하지 않는다는 특징입니다.
  - `HTTP 1.1버전의 특징이 무엇인가요?`
    HTTP 1.1의 큰 특징은 persistent connection과 pipe lining입니다.
    persistent connection는 요청 - 응답을 주고 받았을 때 연결을 종료하지 않고 유지하는 방식입니다.
    같은 클라이언트와 서버가 계속 통신할 일이 많아지자 요청 - 응답이 끝나도 연결이 끊어지지 않는 keep alive가 표준화 되었습니다.
    pipe lining은 한 번에 여러 개의 요청을 보내고 요청 순서대로 응답이 오는 방식입니다.
    - `HOLB가 무엇인가요?`
      HOLB는 head of line blocking의 약자로 시간이 오래 걸리는 요청에 의해 다음 요청이 지연되는 것을 말합니다.
    - `HTTP 2.0버전의 특징은 무엇인가요?`
      HTTP 2.0의 큰 특징은 바이너리 프레이밍과 멀티 플렉싱이 있습니다.
      바이너리 프레이밍은 패킷을 프레임으로 나누고, 이를 binary인코딩하는 방식입니다. 시작줄과 헤더를 HEADERS 프레임에, 바디를 DATA 프레임에 담습니다.
      멀티 플렉싱은 여러 요청을 보내고, 동시에 여러 요청을 받는 형태입니다. 연속적으로 데이터를 받는 스트림을 활용해서 요청된 순서와 상관없이 프레임 단위로 응답을 받습니다.
      하지만 여전히 TCP를 사용하기 때문에 HOLB가 발생할 수 있다는 문제저은 해결되지 않았습니다.
    - `HTTP 3.0의 특징은 무엇인가요?`
      HTTP 3.0은 UDP를 기반으로 만들어졌습니다. 기존에 사용하던 TCP는 데이터의 신뢰성, 순서를 보장하기 위한 로직을 사용하기 때문에 무겁습니다. 시간이 지나면서 모바일 기기를 활용한 단발적인 연결이 많아졌기 때문에 이런 로직이 없는 UDP에 신뢰 보장 로직을 추가한 Quic을 개발하였습니다.
      HTTP 3.0은 UUID를 통해 사용자를 식별합니다. 클라이언트와 연결되면 UUID를 캐싱하고, 이미 캐싱되어 있다면 연결을 생략하고 바로 패킷을 전송합니다.
  - HTML이 무엇인가요?
    HTML은 Hyper Text Markup Language의 약자로 1차원적인 텍스트를 중첩해서 다차원으로 구조화한 것을 markup언어로 표현한 것이다.
  - XML과 HTML은 무엇이 다른가요?
    XML은 데이터의 저장과 전달을 목적으로 하는 마크업 언어로 태그를 원하는대로 커스터마이징해서 사용할 수 있습니다.
    HTML은 웹페이지의 구조와 시각적인 표현을 목적으로 하는 마크업 언어로 태그의 역할이 정해져있습니다.
  - Markup과 Markdown의 차이는 무엇인가요?
    Markup 언어는 문서나 데이터를 태로 구조화하는 언어입니다. Markdown은 사용자가 읽고 쓰기 쉽게 경량화된 Markup언어의 한 종류입니다.
