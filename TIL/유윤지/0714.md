## 🧠 상세 내용 정리

### 1. [주제1] JWT란?

- 디지털 증명서 : 사용자 신원 증명
  서버가 로그인 성공 시 토큰 발급, 클라이언트는 해당 토큰을 계속 들고 다니며 인증
- 구조<br/>
  Header, Payload, Signature<br/>
  `.`으로 각각을 구분해 하나의 문자열로 만듦<br/>
  `e.g.) add.bbb.ccc`
- 역할<br/>
    - Header : JWT(Token Type)와 서명 알고리즘 정보
    - Payload : 사용자 정보(userId), 만료시간 등 실제 데이터
    - Signature : 위/변조 방지용, 서버 비밀키로 생성

### 2. [주제2] JWT vs HTTP/Session

| **구분**    | **HTTP Protocol**   | **Session-Based Authentication** |
|-----------|---------------------|----------------------------------|
| **상태 유지** | 무상태(Stateless)      | 상태 유지(Stateful)                  |
| **인증 정보** | 요청마다 인증 필요          | 서버가 세션으로 인증 상태 저장                |
| **저장 위치** | 서버는 클라이언트 상태 저장 안 함 | 서버에 세션 데이터 저장                    |
| **식별 방법** | 요청 자체에는 사용자 정보 없음   | 세션 ID(쿠키 등)로 사용자 식별              |
| **보안**    | 인증 정보 노출 위험 있음      | 인증 정보는 서버에만 저장, 상대적으로 안전         |
| **확장성**   | 단순, 서버 부담 적음        | 서버가 세션 관리, 대규모 확장 시 부담           |
| **사용 예시** | 공개 웹사이트, REST API 등 | 로그인/마이페이지 등 사용자 인증 필요 서비스        |


| **항목**       | **세션(Session)** | **토큰(Token, JWT)**       |
|--------------|-----------------|--------------------------|
| 인증 정보 저장     | 서버(메모리/DB)      | 클라이언트(토큰 자체)             |
| 인증 식별        | 세션 ID(쿠키)       | JWT 등 토큰(헤더)             |
| 서버 확장        | 세션 동기화 필요       | 별도 동기화 불필요               |
| 로그아웃 처리      | 서버에서 세션 삭제      | 토큰 만료 전까지는 무효화 어려움       |
| REST API 적합성 | 낮음              | 높음                       |
| 민감 정보 노출     | 서버에만 저장         | 토큰에 직접 담으면 위험            |
| 관리 복잡도       | 서버 관리 필요        | 클라이언트 관리 필요              |
| 대표 사용 예시     | 전통적 웹 서비스       | 모바일, SPA, Microservice 등 |
