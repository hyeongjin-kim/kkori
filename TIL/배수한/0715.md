## Authorization Code Grant 흐름

1. **리소스 오너(사용자)가 로그인 요청**
    - 사용자가 "카카오 로그인" 등의 소셜 로그인 버튼 클릭
2. **클라이언트가 Authorization URL 생성 → 사용자를 Authorization 서버로 리디렉트**
    - URL에 다음 정보 포함:
        - `client_id` (앱의 ID)
        - `redirect_uri` (인증 완료 후 돌아올 URL)
        - `response_type=code` (Authorization Code Flow이므로)
        - 필요 시 `scope`, `state` 등도 추가
3. **사용자가 Authorization 서버에서 로그인 후, Authorization Code를 redirect_uri로 전달**
    - 예시:
        
        ```
        https://example.com/oauth/**callback**?code=abc123
        ```
        
4. **클라이언트가 redirect_uri에서 코드 수신 → Access Token 요청**
    - 토큰 요청 시 포함하는 것:
        - `grant_type=authorization_code`
        - `client_id`
        - `redirect_uri` (처음 보낸 것과 동일해야 함)
        - `code` (Authorization 서버가 보낸 코드)
5. **토큰 발급 후, 사용자에게 전달**

---

## 🤔 느낀 점 & 회고

오늘 OAuth Authorization Code Grant 흐름을 정리하면서, 그동안 모호하게 알고 있었던 로그인 과정의 전체적인 구조를 보다 명확하게 이해할 수 있었습니다. 단순히 "소셜 로그인 버튼을 누르면 로그인된다"는 식으로만 알고 있던 흐름이, 실제로는 클라이언트, 인증 서버, 리소스 서버 간의 역할 분리와 토큰 발급 절차를 통해 체계적으로 이루어진다는 점이 인상 깊었습니다.

특히 클라이언트가 Authorization Code를 받은 뒤 Access Token을 요청할 때 redirect_uri가 정확히 일치해야 한다는 점이나, 발급받은 토큰을 통해 사용자 정보를 어떻게 활용할 수 있는지를 학습하면서 보안 측면에서도 많은 것을 배울 수 있었습니다.
